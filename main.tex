\section{Test setup}

The laptop I used for testing is a M3 Macbook Air running macOS 26.2.
For creating the test plans and performing the performance tests I have used Apache JMeter.

\section{Test Plan}
The System Under Test I decided to use for my tests was a sample test website provided by BlazeMeter: \url{https://blazedemo.com/}.
In my tests I have targeted the homepage, the reserve page and the purchase page. The URLs for the pages are the following:
\begin{itemize}
  \item Homepage: \url{https://blazedemo.com/index.php}
  \item Reserve page: \url{https://blazedemo.com/reserve.php}
  \item Purchase page: \url{https://blazedemo.com/purchase.php}
\end{itemize}

\section{Initial Expectations}

Before running the tests I expected the site to respond within 500 - 1000 milliseconds during the baseline performance test. I also expected the stress test, and spike tests both to cause about a 100 milliseconds increase in the median response times.

The results are given for each of the requests and the aggregate total.

For each of the test types I have performed 3 separate runs and will present the results for each of them.
The three runs were made on different days, during different time of the day.

\section{Conclusion}
As we can see in the results, the System Under Test responded during the baseline  performance tests close to the expected response times, with median response times 
ranging between 397 and 531 milliseconds across the three runs.

The stress test produced results similar to our baseline tests, which goes against what I initially expected. The median response times during the stress 
test runs ranged between 393 and 547 milliseconds, showing only a slight increase compared to the baseline. 

However, the spike test produced noticeably slower response times across all three runs, as expected. 
The median response times during the spike runs were 530, 673, and 681 milliseconds respectively, compared to a baseline median of roughly 400-531 milliseconds. 

Also more importantly, the average response time was 105 milliseconds higher than the median response time in the spike test, which seems to indicate that in the measurements we had a few slower response times.
Also, the maximum response times during the spike test were 1850, 5699, and 4492 milliseconds across the three runs, indicating that under sudden high load the server occasionally struggled to respond in a timely manner.  

Also notably, the System Under Test hasn't produced any errors throughout the total 9 test runs as we had a 0.00\% error rate in all of nine of them.
